# icegres Scorecard

Definitive parity + performance record for **icegres** — a serve-in-place
Iceberg lakehouse server speaking the Postgres wire protocol — measured
against the behavioral bar set by Databricks Lakebase (= Neon architecture)
and Moonlink. Contract: `bench/SPEC.md`. All raw probe/benchmark JSONs live
in `bench/results/`; the parity matrix below is machine-regenerated by
`bench/parity.sh`, benchmark tables are appended by `bench/bench.sh`, and
every improvement landed through the regression gate `bench/gate.sh`
(FAIL if any latency p50 worsens >20%, qps drops >10%, rss grows >25%,
binary grows >10%, e2e not green, or any parity verdict downgrades —
memory and binary size are first-class gated metrics as of round 5).

## Reproduction (3 commands)

```bash
bash infra/scripts/up.sh    # 1. start/health-check the stack (Postgres :5433, RustFS S3 :9000, Lakekeeper REST :8181)
bash bench/parity.sh        # 2. all 25 parity probes -> bench/results/parity-<ts>.json + regenerates the matrix below
bash bench/bench.sh         # 3. full benchmark      -> bench/results/bench-<ts>.json  + table appended below
```

Regression gate for landing a change: `bash bench/gate.sh` (compares a
fresh bench run against `bench/results/baseline.json`, runs e2e, checks
parity no-downgrade). Memory-bound micro-check for the time-travel cache:
`bash icegres/tests/pinned_bound.sh`.

Correctness suite (also enforced by `bench/gate.sh`):
`bash icegres/tests/e2e.sh` — 19 assertions. Note: every e2e/parity run
appends a few rows (= small Parquet files) to the append-only `demo.trips`;
see **Layout drift** at the bottom before comparing full-scan numbers
across runs.

## GAP roadmap — the 6 honest gaps

All six are consequences of two facts: the pgwire front end ships no-op
auth/plaintext transport by default, and the pinned iceberg-rust 0.9.1 /
iceberg-datafusion integration is append-only (fast_append is the only
data-changing transaction action). None can be fixed by icegres-side code
alone without either real security plumbing or an upstream Iceberg
write-path release; each has a concrete roadmap:

| id | gap | why it exists | roadmap |
|----|-----|---------------|---------|
| A6 | no server-side auth | `AuthManager::default()` is a no-op; any user/password connects | icegres already runs its own accept loop (`icegres/src/ops.rs`, built for idle-shutdown); wire pgwire's SCRAM/cleartext `StartupHandler` there against a password file / env secret. Effort: small-medium, no upstream blocker. |
| A7 | no TLS | no rustls acceptor on the listener | short term: terminate TLS in front (stunnel/haproxy/pgbouncer). Native: pgwire supports rustls; plumb cert/key config into the same ops.rs accept loop. No upstream blocker. |
| B2 | UPDATE rejected | iceberg-datafusion 0.9 is append-only; Iceberg mutation needs v2 delete files (MoR) or a rewrite-files action (CoW) — neither exists in 0.9.1 | blocked on upstream iceberg-rust; revisit at each release. Do NOT hand-forge REST commits outside the library's conflict detection. |
| B3 | DELETE rejected | same as B2 | same as B2. |
| B4 | BEGIN/COMMIT/ROLLBACK accepted but non-transactional (INSERT inside ROLLBACK persists) | every statement auto-commits an Iceberg snapshot | feasible within 0.9.1 for INSERT-only transactions: buffer appends per session, emit ONE `fast_append` commit at COMMIT, drop buffers at ROLLBACK. Medium effort; honest interim behavior documented by the probe. |
| B5 | no PK/unique constraints (duplicate insert accepted) | Iceberg has no constraint concept | enforcement would need a read-before-write existence check per INSERT — expensive and racy across multiple computes (D2). Documented as an architectural anti-goal; dedupe at query time or in future compaction. |

N/A-BY-DESIGN (1): C3 CDC Postgres→Iceberg — Moonlink exists to copy
Postgres heap data into Iceberg; icegres data is *born* in Iceberg (each
wire INSERT is an Iceberg commit, proven by B1+C1), so there is no second
copy to synchronize. Parity by construction.

### Layout drift — read this before comparing runs

`demo.trips` is append-only and every e2e/parity run adds rows as new small
Parquet files (~1 row/file). Each extra file costs ~0.3–1 ms per full scan.
The mechanism was proven during parity-features acceptance: a **control
binary with the change removed showed the same "regression" vs baseline**
(qps −23.5%, aggregate +28.2% at 4 extra files) while candidate-vs-control
at matched layout showed none; the same control-at-same-clock method
cleared tokio-trim-footprint in round 5. Since round 5 the bench harness
canonicalizes the layout (re-seeds `demo.trips` when drifted) before
measuring, so bench-vs-bench comparisons are layout-stable; e2e/parity
still append between runs. Observed qps noise across same-code runs
reaches ±34% in the worst historical window (mitigated to low single
digits by median-of-3 windows); rss_peak carries a few-percent band from
sampling alignment. Gate rule of thumb: if a gate fails on qps/aggregate
only, benchmark a control build at the same table layout and wall-clock
before blaming the change. Reset path: drop the demo tables via Lakekeeper
REST (`purgeRequested=true`) and re-run `icegres seed` (restores 1 file
per table; documented in `icegres/src/seed.rs`).


## Detailed improvement notes

### 1. metadata-caching (accepted; baseline `163255Z` → `164622Z`)

**Change** (`icegres/src/cache.rs`, wired in `icegres/src/context.rs`):
snapshot-aware table metadata caching. iceberg-datafusion's catalog-backed
provider called `load_table` on every scan and rebuilt a `Table` whose
internal manifest cache starts cold, so every query re-read the manifest
list + all ~231 manifest avro files from RustFS (~1 ms/GET ≈ the entire
~220 ms point-lookup p50); the Lakekeeper REST call itself is only
~2–3 ms. `CachingTableProvider` keeps the last-seen `Table` (warm moka
manifest cache) inside an `IcebergStaticTableProvider` and does one cheap
REST `load_table` per scan purely to compare metadata-location/snapshot-id:
unchanged → reuse warm provider; changed (commit from this server or any
other writer) → rebuild. **Freshness stays exact — no TTL, zero staleness
window.** INSERTs delegate to the upstream provider; metadata tables and
DDL fall through.

Gate PASS (e2e green; parity `164753Z` unchanged at 16/8/1):
point_lookup 221.64 → 14.74 (−93.3%), filtered_scan 230.12 → 14.95
(−93.5%), aggregate 253.96 → 121.56 (−52.1%), join 257.80 → 127.39
(−50.6%), qps 7.6 → 28.1 (+269.7%); writes/freshness within gate
(+9.7…+12.8%, noise).

### 2. datafusion-tuning (accepted; baseline `164622Z` → `171214Z`)

**Measurement first**: interleaved A/B sweeps proved every classic
SessionConfig/RuntimeEnv knob a no-op on this workload; `EXPLAIN ANALYZE`
showed all operator time in µs with ~all latency in the single-partition
`IcebergTableScan` stream, whose object-store IO is hardwired to
`num_cpus` (= 4) concurrent GETs in iceberg-rust 0.9. The effective
"object-store concurrency" knob lives in the Iceberg scan builder, not
SessionConfig.

**Change**: `icegres/src/scan.rs` — `TunedIcebergScan`, a drop-in
`ExecutionPlan` rebuilt from the upstream scan's public getters, running
the identical `TableScan` with `.with_concurrency_limit(N)`;
`N` from `ICEGRES_SCAN_CONCURRENCY` (default 32; `0` = upstream behavior).
Wired in `CachingTableProvider::scan`; inserts/metadata/DDL untouched.
Plus `ICEGRES_DF_OPTS` env passthrough for ops tuning (defaults unchanged
— nothing measured an improvement).

Gate PASS (e2e green; parity `171352Z` unchanged at 16/8/1):
point_lookup −20.8%, filtered_scan −18.0%, aggregate −25.8%, join −21.8%,
freshness −14.3%, qps 0% (8 conns already saturate 4 cores), nothing
regressed. Remaining read cost diagnosed as file-count-bound (~236 tiny
Parquet files) → next improvement.

### 3. seed-file-layout (accepted; baseline `171214Z` → `172127Z`)

**Change** (`icegres/src/seed.rs`, `icegres/src/context.rs`): seed writes
each demo table as exactly ONE Parquet file in ONE Iceberg commit —
`insert_all` (single 280-row INSERT) instead of 100-row chunks, and the
seed session pins `target_partitions = 1`
(`build_session_context_with(catalog, Some(1))`; iceberg-datafusion writes
one file per non-empty repartition worker, so 1 partition ⇒ 1 file
deterministically). serve/sql paths unchanged. Demo tables were dropped
(Lakekeeper REST, `purgeRequested=true`) and re-seeded: `demo.trips` went
from 239 data files to 1. No `icegres compact` — see "Rejected along the
way".

Gate PASS (e2e green; parity `172254Z` unchanged at 16/8/1):
aggregate −92.9% (90.18 → 6.39), join −90.5%, filtered_scan −52.0%,
point_lookup −45.5%, qps +1533.8% (28.1 → 459.1), rss −10.8%, writes
−6/−9%. Read metrics are intentionally not layout-comparable across the
re-seed — the layout change IS the improvement.

### 4. parity-features (accepted; baseline `172127Z` → `173613Z`, parity baseline → `174423Z`)

**Change** — closed the two cheap parity GAPs and hardened E2:

- **D5 scale-to-zero** (`icegres/src/ops.rs`): `icegres serve
  --idle-shutdown-secs N` (env `ICEGRES_IDLE_SHUTDOWN_SECS`) exits cleanly
  (code 0) after N seconds with zero client connections, via an own accept
  loop over `pgwire::tokio::process_socket` (flagless path = stock
  datafusion-postgres loop, byte-for-byte). Probe: served queries, exited
  on its own 5.0 s after the last connection, respawned and answered in
  ~1.3 s → GAP → PASS. Supervisor pattern documented in ops.rs + README.
- **D4 time travel** (`icegres/src/cache.rs`): quoted
  `"table@snapshot_id"` references resolve to a read-only provider pinned
  via `IcebergStaticTableProvider::try_new_from_table_snapshot` (immutable
  → cached forever; unknown ids error; INSERT rejected). Probe: pinned
  oldest snapshot returns 280 rows vs current live count → GAP → PASS.
- **E2 health** (`icegres/src/ops.rs`): `--health-port P` /
  `ICEGRES_HEALTH_PORT` — minimal HTTP liveness endpoint (200 "ok");
  health pings deliberately do not reset the idle clock.

Parity 16/8/1 → **18/6/1**, zero downgrades; e2e green. First gate run
FAILed vs the stale baseline — proven environmental by control-binary A/B
(see "Layout drift"); candidate vs control at matched layout: aggregate
−13.4%, filtered_scan −9.0%, join −7.7%, qps +11.5%, nothing regressed.
Two candidate runs agreed within 25% on every metric; `173613Z` became
`baseline.json`.

### 6. tokio-trim-footprint (accepted; baseline `182535Z` → `192303Z`) — 2026-07-05

**Change** (`icegres/Cargo.toml`, `bench/harness/Cargo.toml`) — resource
hygiene, no source code touched:

- **tokio `"full"` → 5 used features** (`rt-multi-thread`, `macros`, `net`,
  `io-util`, `time`): drops `process`, `signal`, `signal-hook-registry`,
  `io-std` from the compiled feature union (`fs`/`sync` stay — transitive
  deps require them). Harness tokio was already minimal.
- **clap default features off** (keep `std help usage error-context derive
  env`): drops `color` (anstream/anstyle) and `suggestions` (strsim).
- **`[profile.release] strip = "symbols"`** in icegres and harness — the
  bulk of the win; rust symbol tables on the datafusion+iceberg graph are
  enormous.

**Resource deltas vs baseline (the point of this round):**

### 7. bounded-caches (accepted; baseline `192303Z` → `195716Z`) — 2026-07-05

**Change** (`icegres/src/cache.rs` only): the time-travel pinned-provider
map (`table@snapshot_id` → `IcebergStaticTableProvider`, added by
improvement 4/D4) cached every distinct snapshot ever queried, forever.
Each pinned provider holds a full `Table` — metadata JSON, schema, warm
moka manifest cache — so a client sweeping history (a dashboard iterating
`trips$snapshots`, or an adversary issuing distinct `AS OF` reads) grew
server RSS without bound, ~0.5 MB per distinct snapshot on the tiny demo
table (more on real tables with bigger manifests). The map is now a
bounded per-table LRU: at most **16 pinned snapshots per base table**
(`MAX_PINNED_PER_TABLE`), least-recently-used evicted on insert, LRU
ticks maintained lock-free (relaxed `AtomicU64` under the existing read
lock — the hot-path lock structure is unchanged). Evictions and cache
size are logged at `debug` for observability. An evicted snapshot stays
fully queryable — re-querying it just pays the one-time provider rebuild
(~1 REST `load_table` + manifest reads, the same cost its *first* query
always paid). The current-snapshot cache (`CachingTableProvider`) was
audited too: it holds exactly one provider per table, bounded by table
count — no change needed.

**Proof of bound** (new `icegres/tests/pinned_bound.sh`, port 5444,
self-contained, non-destructive — its own scratch table, dropped after):
commits 56 snapshots, time-travel-queries **all 56 distinct snapshots**
(row counts verified per snapshot), then asserts three ways:
max observed pinned size **16 = cap**, evictions **40 = 56 − 16**, RSS
growth from cache-full to end of sweep **+6.2 MB < 50 MB limit**
(75.7 → 81.9 MB). Control with the pre-fix binary at the same wall-clock,
same sweep: **+19.4 MB** (80.0 → 99.4 MB) and linear in snapshot count —
the bound is real and the check would catch a regression. Plus 7 unit
tests on the LRU policy (`cargo test`: eviction order, per-table scoping,
recent-use survival, 100-snapshot churn never exceeds cap).

**Perf / resource trade-off** (gate `baseline.json` = `192303Z` →
`195716Z`, PASS on every rule; e2e green; parity `195848Z` = 18 PASS /
6 GAP / 1 N/A, zero downgrades):

| metric | baseline | candidate | delta | note |
|--------|----------|-----------|-------|------|
| point_lookup_ms p50 | 6.74 | 5.79 | −14.1% | drift-favorable; claim is "no regression", not a win |
| filtered_scan_ms p50 | 6.43 | 5.60 | −12.9% | " |
| aggregate_ms p50 | 6.58 | 5.94 | −9.7% | " |
| join_ms p50 | 9.85 | 8.41 | −14.6% | " |
| insert/freshness p50 | 48.36 / 56.94 | 48.79 / 59.03 | +0.9% / +3.7% | within run noise |
| qps_8conn | 445.4 | 448.9 | +0.8% | flat |
| rss_idle_mb | 75.74 | 76.15 | +0.5% | flat |
| rss_peak_mb | 90.6 | 92.36 | +1.9% | flat (±34% qps-window noise band applies) |
| binary_size_mb | 120.0 | 120.01 | +0.01 MB | LRU code is ~100 lines |

The read-latency improvements are environmental (this change adds code to
the time-travel path only, which the bench never exercises); the honest
claim is **zero cost on the normal path** for a hard memory bound on the
adversarial one. Cap choice: 16 comfortably covers interactive time-travel
use (a session comparing a handful of snapshots) while capping worst-case
pinned memory at 16 × per-provider footprint per table; it is a compile-time
constant, trivially revisited if real usage wants more.
