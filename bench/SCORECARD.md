# icegres Scorecard

Definitive parity + performance record for **icegres** ‚Äî a serve-in-place
Iceberg lakehouse server speaking the Postgres wire protocol ‚Äî measured
against the behavioral bar set by Databricks Lakebase (= Neon architecture)
and Moonlink. Contract: `bench/SPEC.md`. All raw probe/benchmark JSONs live
in `bench/results/`; the parity matrix below is machine-regenerated by
`bench/parity.sh`, benchmark tables are appended by `bench/bench.sh`, and
every improvement landed through the regression gate `bench/gate.sh`
(FAIL if any latency p50 worsens >20%, qps drops >10%, rss grows >25%,
binary grows >10%, e2e not green, or any parity verdict downgrades ‚Äî
memory and binary size are first-class gated metrics as of round 5).

**Bottom line (2026-07-06, R8+R9): 27 PASS / 0 GAP / 1 N/A-BY-DESIGN.**
Every behavioral gap to the bar is closed ‚Äî auth, TLS, UPDATE, DELETE,
transactions, PK enforcement, plus opt-in Moonlink-style buffered writes,
Neon-style writable zero-copy branches, the `icegresd` control plane
(wake-on-connect scale-to-zero, one-port branch-endpoint routing,
supervised computes: D5 upgraded + new D7), verified ORM/driver
compatibility (new A8: SQLAlchemy/psycopg2/pg8000/pandas), and icegresd
warm session pooling (connect ‚Üí ReadyForQuery 0.44 ms p50 via the proxy,
~165√ó faster than a 73 ms wake-after-idle) ‚Äî with a single copy of the
data in open Iceberg format, no replication, no compromise on default
durability/atomicity, on the pinned crate matrix, and with reads
‚àí67‚Ä¶‚àí72% p50 / qps +357% / binary ‚àí18.5% vs the session's original state.
Confirmed by the round-8 wrap fresh verification pass: e2e **104/104**,
parity `035002Z` **27/0/1**, bench `035052Z` gate **PASS on all 13 rules**
vs `baseline.json` (= `033809Z`). Recommended production topology
(OLTP + API + BI tiers, when-to-use table, anti-patterns):
**`docs/cqrs-topology.md`**.

Feature-mode examples (each is exactly what the harnesses probe):

```bash
# auth + TLS (A6/A7)
bash infra/scripts/gen-dev-cert.sh
printf 'app_user:s3cret\n' > /tmp/auth.conf
icegres serve --auth-file /tmp/auth.conf \
  --tls-cert infra/.data/tls/dev.crt --tls-key infra/.data/tls/dev.key &
PGPASSWORD=s3cret psql "host=localhost port=5439 user=app_user dbname=icegres \
  sslmode=verify-full sslrootcert=infra/.data/tls/dev.crt" -c 'select 1'

# transactions + PK enforcement (B4/B5)
icegres serve --enforce-pk &   # tables with the icegres.primary-key property get 23502/23505
psql -h 127.0.0.1 -p 5439 -c 'begin; insert into demo.trips values (...); commit;'

# Moonlink-style buffered writes (opt-in; <=N ms durability window, WARN on enable)
icegres serve --write-buffer-ms 100 &   # INSERT acks ~1.5 ms; same-server reads union the buffer

# Neon-style zero-copy branches (D6)
icegres branch create demo.trips dev          # ONE metadata commit, zero data copied
icegres serve --branch dev --port 5440 &      # second endpoint, writes isolated from main
icegres branch drop demo.trips dev            # removes only the ref

# icegresd control plane (D5 full cycle + D7 routing/supervision)
icegresd serve --port 5432 --idle-shutdown-secs 300 &  # computes spawn on connect, idle-exit to zero
psql -h 127.0.0.1 -p 5432 -d icegres -c 'select 1'     # wakes the main compute (~73 ms after idle)
psql -h 127.0.0.1 -p 5432 -d 'icegres@dev'             # routes to a per-branch compute
icegresd status                                        # computes, ports, PIDs, restart counts, pool stats

# icegresd session pooling (default --pool-size 8): warm pre-handshaked backend
# conns; matching clients reach ReadyForQuery in <1 ms (connect_via_proxy_ms).
# SESSION pooling only ‚Äî one client per backend conn, never reused, and no
# transaction pooling (session state: txn buffers/SET/prepared statements).
# The pool idle-drains after --pool-idle-secs so scale-to-zero still works.
for i in 1 2 3; do psql -h 127.0.0.1 -p 5432 -d icegres -c 'select 1'; done  # warm handouts

# ADBC first-class (A11): Arrow Flight SQL endpoint + bulk ingest
icegres flight-serve &                        # grpc://0.0.0.0:50051, same engine wiring
python3 - <<'PY'
import adbc_driver_flightsql.dbapi as fs, pyarrow as pa
conn = fs.connect("grpc://127.0.0.1:50051"); cur = conn.cursor()
cur.execute("SELECT count(*) FROM demo.trips"); print(cur.fetchone())
cur.adbc_ingest("adbc_ingest", pa.table({...}), mode="append", db_schema_name="demo")
PY
# postgres-lane ADBC reads work against plain `icegres serve` too:
# adbc_driver_postgresql wraps queries in COPY (‚Ä¶) TO STDOUT (FORMAT binary),
# which icegres answers on both wire protocols (use autocommit=True).
python3 bench/clients/a11_adbc_probe.py   # both lanes, pass/xfail per capability
```

Postgres heap data into Iceberg; icegres data is *born* in Iceberg (each
wire INSERT is an Iceberg commit, proven by B1+C1), so there is no second
copy to synchronize. Parity by construction ‚Äî the user directive "single
copy in Iceberg, everything zero-copy on the lake" makes a CDC pipeline
definitionally unnecessary.

How each gap was closed (all on the pinned crate matrix ‚Äî nothing bumped,
nothing vendored):

- **A6 auth (R6.1)** ‚Äî `--auth-file` (env `ICEGRES_AUTH_FILE`) enables
  SCRAM-SHA-256 via pgwire's SASL handler against a `user:password` file
  (`icegres/src/pgauth.rs`; salted hashes in memory, wrong password /
  unknown user ‚Üí FATAL 28P01; no file = historical permissive mode +
  startup WARN).
- **A7 TLS (R6.1)** ‚Äî `--tls-cert`/`--tls-key` (env `ICEGRES_TLS_CERT/KEY`)
  put a rustls acceptor on the listener; any TLS misconfiguration aborts
  boot (no silent plaintext fallback, unlike upstream datafusion-postgres);
  dev certs via `infra/scripts/gen-dev-cert.sh`. Like stock Postgres the
  listener still accepts plaintext startup, so the A7 probe requires
  `sslmode=require` + `verify-full` + an `openssl s_client -starttls
  postgres` handshake as evidence.
- **B2 UPDATE + B3 DELETE (R6.2)** ‚Äî real copy-on-write DML with a single
  data copy on the lake (`icegres/src/overwrite.rs` + `dml.rs`): icegres
  classifies every live data file against the predicate, rewrites ONLY the
  files containing matching rows (all others are reused by reference), and
  commits one overwrite snapshot through the Iceberg REST protocol with an
  `assert-ref-snapshot-id` requirement (409 conflict ‚Üí bounded
  recompute-and-retry, proven against Lakekeeper by fault injection).
  Time travel over pre-DML snapshots keeps working. The earlier "do NOT
  hand-forge REST commits" caution is resolved, not ignored: conflict
  detection happens server-side via the REST requirement ‚Äî the same
  optimistic-concurrency mechanism the library's own `Transaction` uses.
- **B4 transactions (R6.3)** ‚Äî real BEGIN/COMMIT/ROLLBACK
  (`icegres/src/txn.rs`): snapshot-pinned reads per table (snapshot
  isolation), session-buffered writes with read-your-own-writes overlays,
  COMMIT composes everything into ONE Iceberg snapshot anchored at the pin
  (first-committer-wins, 40001 on conflict ‚Äî no blind retry), statement
  errors poison the txn (25P02).
- **B5 constraints (R6.3)** ‚Äî opt-in `--enforce-pk` + table property
  `icegres.primary-key`: NOT NULL (23502) + uniqueness (23505) checks
  validated against the very snapshot each commit anchors to, including a
  transaction's own buffered rows. Off by default (documented cost: reads
  key columns of every live file per write).
- **D6 zero-copy branches (R6.5, new probe)** ‚Äî Neon's branch-per-endpoint
  model over Iceberg snapshot refs (`icegres/src/branch.rs`): `icegres
  branch create/list/drop` (create/drop = ONE metadata commit, zero data
  copied) + `icegres serve --branch <name>` (reads pin to the branch head,
  ALL writes commit with `assert-ref-snapshot-id` on the branch ref ‚Äî
  isolation from `main` proven bidirectionally in e2e and the D6 probe).
- **D7 endpoint routing + supervised computes (R7, new probe)** ‚Äî the
  `icegresd` control plane (`icegres/src/bin/icegresd.rs`, a second ~3 MB
  binary): a pgwire-aware proxy that parses ONLY the plaintext
  StartupMessage `database` parameter, wakes the target compute on connect
  (spawn `icegres serve --idle-shutdown-secs`, poll TCP readiness, forward
  the ORIGINAL startup bytes, `copy_bidirectional` splice), routes
  `icegres@<branch>` to per-branch computes on ephemeral localhost ports,
  reaps idle-exited computes, and restarts UNCLEAN exits with capped
  exponential backoff (0.5/1/2 s, max 3 per crash episode). `icegresd
  status` reports computes/branches/ports/PIDs/restart counts from a JSON
  status file. TLS terminates at the compute (proxy answers SSLRequest
  with `N`, plain TCP to computes on localhost); CancelRequest is not
  routed (no backend-key tracking, documented).

## Detailed improvement notes

### 1. metadata-caching (accepted; baseline `163255Z` ‚Üí `164622Z`)

**Change** (`icegres/src/cache.rs`, wired in `icegres/src/context.rs`):
snapshot-aware table metadata caching. iceberg-datafusion's catalog-backed
provider called `load_table` on every scan and rebuilt a `Table` whose
internal manifest cache starts cold, so every query re-read the manifest
list + all ~231 manifest avro files from RustFS (~1 ms/GET ‚âà the entire
~220 ms point-lookup p50); the Lakekeeper REST call itself is only
~2‚Äì3 ms. `CachingTableProvider` keeps the last-seen `Table` (warm moka
manifest cache) inside an `IcebergStaticTableProvider` and does one cheap
REST `load_table` per scan purely to compare metadata-location/snapshot-id:
unchanged ‚Üí reuse warm provider; changed (commit from this server or any
other writer) ‚Üí rebuild. **Freshness stays exact ‚Äî no TTL, zero staleness
window.** INSERTs delegate to the upstream provider; metadata tables and
DDL fall through.

### 7. bounded-caches (accepted; baseline `192303Z` ‚Üí `195716Z`) ‚Äî 2026-07-05

**Change** (`icegres/src/cache.rs` only): the time-travel pinned-provider
map (`table@snapshot_id` ‚Üí `IcebergStaticTableProvider`, added by
improvement 4/D4) cached every distinct snapshot ever queried, forever.
Each pinned provider holds a full `Table` ‚Äî metadata JSON, schema, warm
moka manifest cache ‚Äî so a client sweeping history (a dashboard iterating
`trips$snapshots`, or an adversary issuing distinct `AS OF` reads) grew
server RSS without bound, ~0.5 MB per distinct snapshot on the tiny demo
table (more on real tables with bigger manifests). The map is now a
bounded per-table LRU: at most **16 pinned snapshots per base table**
(`MAX_PINNED_PER_TABLE`), least-recently-used evicted on insert, LRU
ticks maintained lock-free (relaxed `AtomicU64` under the existing read
lock ‚Äî the hot-path lock structure is unchanged). Evictions and cache
size are logged at `debug` for observability. An evicted snapshot stays
fully queryable ‚Äî re-querying it just pays the one-time provider rebuild
(~1 REST `load_table` + manifest reads, the same cost its *first* query
always paid). The current-snapshot cache (`CachingTableProvider`) was
audited too: it holds exactly one provider per table, bounded by table
count ‚Äî no change needed.

### 8. auth-tls (accepted; baseline `195716Z` ‚Üí `203036Z`, parity baseline `200300Z` ‚Üí `203811Z`) ‚Äî 2026-07-05

Closes the two security gaps, A6 (server-side auth) and A7 (TLS), exactly
along the roadmap lines above: both features ride the accept loop icegres
already owns (`ops.rs::serve_custom`, the generalization of the
idle-shutdown loop).

**What shipped**

- `--auth-file <path>` (env `ICEGRES_AUTH_FILE`): `user:password` lines
  (`#` comments; username may not contain `:`, password may; duplicate
  users and empty files are hard startup errors). Auth runs SCRAM-SHA-256
  (RFC 5802/7677) via pgwire's SASL handler (`icegres/src/pgauth.rs`
  implements `AuthSource`): the cleartext never crosses the wire, the
  server holds only `Hi(password, salt, 4096)` with a per-user random
  16-byte `/dev/urandom` salt, and wrong password / unknown user both fail
  with Postgres-standard `28P01` (same error shape ‚Äî no username oracle).
  No `--auth-file` = the historical permissive mode, now with a startup
  WARN so the default is explicit, never silent.
- `--tls-cert`/`--tls-key` (env `ICEGRES_TLS_CERT/KEY`): rustls acceptor
  on the pgwire listener (PKCS#8/RSA/SEC1 PEM). Deliberate divergence from
  upstream: datafusion-postgres `serve_with_handlers` logs a warning and
  serves **plaintext** when TLS setup fails; icegres aborts the boot ‚Äî a
  server asked to be secure never comes up insecure. Like stock Postgres
  (and upstream pgwire) the TLS listener still accepts plaintext startup;
  clients enforce encryption with `sslmode=require`/`verify-full`, which is
  why the A7 probe demands `sslmode=require` AND `verify-full` against the
  pinned cert AND an `openssl s_client -starttls postgres` handshake
  (TLSv1.3, TLS_AES_256_GCM_SHA384 observed) as evidence.
- `infra/scripts/gen-dev-cert.sh`: idempotent self-signed dev cert
  (CN=icegres-dev, SAN localhost/127.0.0.1, PKCS#8) into
  `infra/.data/tls/` (gitignored), key chmod 600.
- Flagless `icegres serve` still takes the stock datafusion-postgres
  loop byte-for-byte ‚Äî the custom loop engages only when idle-shutdown,
  TLS, or auth is requested, so the benched default path is untouched.
- Harnesses: e2e gained section (h) (7 assertions, 19‚Üí26: right password
  over `sslmode=require` + `verify-full`, wrong password rejected, unknown
  user rejected, s_client handshake, authenticated+encrypted data query);
  parity A6/A7 now probe a dedicated `--auth-file`+TLS server on :5447;
  bench/e2e/parity `unset ICEGRES_AUTH_FILE ICEGRES_TLS_*` for their own
  permissive servers so ambient env can never flip what is being measured
  (clients inherit `PGPASSWORD` per psql's normal rules).

### 9. dml-copy-on-write (accepted; baseline `203036Z` ‚Üí `213153Z`, parity baseline `203811Z` ‚Üí `213309Z`) ‚Äî 2026-07-05

Closes B2 (UPDATE) and B3 (DELETE) ‚Äî the "append-only wall" ‚Äî with
**single-copy, zero-replication copy-on-write** on the lake, under the
pinned crate matrix (nothing bumped, nothing vendored).

**Mechanism** (`icegres/src/overwrite.rs`, ~950 lines incl. docs/tests):

- Per statement: load fresh table metadata; for every live data file in
  the current snapshot, read it and evaluate the SQL predicate against its
  rows with an ephemeral DataFusion context (the file's rows registered
  under the real table name, so predicates/aliases resolve verbatim).
  Files with 0 matches are **kept by reference**; a DELETE matching every
  row **removes** the file; anything else is **rewritten** (DELETE
  survivors = `WHERE (pred) IS DISTINCT FROM TRUE` for NULL-correct
  semantics; UPDATE = `CASE WHEN (pred) THEN (expr) ELSE col END`
  projection), with row-accounting invariants that abort the commit on any
  mismatch.
- Replacement rows go through the same iceberg-rust writer stack the
  INSERT path uses (rolling Parquet writer under the table's data
  location). A new manifest records ADDED + EXISTING (original snapshot
  ids/sequence numbers preserved) + DELETED entries; untouched manifests
  are carried into the new manifest list as-is. Snapshot summary totals
  are recomputed exactly from the final live set (iceberg-rust 0.9.1's own
  fast_append writes non-cumulative totals ‚Äî measured and documented).
- Commit = Iceberg REST `POST .../tables/{tbl}` with
  `assert-table-uuid` + `assert-ref-snapshot-id main=<anchor>` and
  `add-snapshot` + `set-snapshot-ref main` updates
  (`CommitTableRequest` from iceberg-catalog-rest, sent with reqwest).
  Concurrency is optimistic and server-checked: on 409 the entire
  computation is redone from fresh metadata, max 3 attempts. Files staged
  by a lost attempt are unreferenced orphans ‚Äî harmless by Iceberg
  semantics. **Proven, not assumed**: `ICEGRES_DML_INJECT_CONFLICT=1`
  sabotages attempt 1's requirement; Lakekeeper answered
  `CatalogCommitConflicts 409 (expected ‚Ä¶777, found ‚Ä¶776)` and attempt 2
  committed ‚Äî asserted in e2e every run.
- Wire-up (`icegres/src/dml.rs`): a datafusion-postgres `QueryHook`
  intercepts exactly `Statement::Update`/`Statement::Delete` on BOTH the
  simple and extended protocols (default `serve` path now uses
  `serve_with_hooks`; the custom accept loop passes the same chain), plus
  `icegres sql`. Everything out of scope ‚Äî joins/USING/RETURNING/LIMIT,
  subqueries in predicates (would see per-file slices), bind parameters,
  partitioned/v1/merge-on-read tables ‚Äî is **rejected loudly** (SQLSTATE
  0A000), never mis-executed. Read-your-writes needs no new invalidation:
  the commit moves `main`, and the per-scan snapshot check in cache.rs
  (round 1) picks it up exactly like INSERT.
- Spike proof on a scratch table (pre-wire-up, `demo.dml_scratch`):
  DELETE/UPDATE verified three ways ‚Äî icegres reads, **pyiceberg 0.11.1**
  independent reads, and time travel to the pre-DML snapshot ‚Äî and
  file-level zero-copy confirmed via pyiceberg scan planning: the
  untouched Parquet file is the *same object* in both snapshots, only the
  file containing matched rows was rewritten.

**Cost/limitations (documented in module docs):** every live data file is
read once per DML statement (no min/max stat pruning yet ‚Äî a future
optimization, not a correctness issue), one file at a time (peak memory ‚âà
one decoded file). DML latency is therefore O(table size); the bench
read/insert metrics are unaffected (hook only fires on UPDATE/DELETE).

**Gate story.** All 13 metrics PASS vs the R6.1 baseline: binary
120.31 ‚Üí 121.56 MB (**+1.0%**, budget +10%), rss_peak 95.39 ‚Üí 92.34 MB
(‚àí3.2%), rss_idle ‚àí2.5%, qps 383.9 ‚Üí **466.0** (+21.4%), point_lookup
12.34 ‚Üí 5.79 ms. Honesty note: the R6.1 baseline was accepted during a
documented box-drift window (its same-clock control scored 376 qps);
today's 466 matches the pre-drift 448.9-qps era, so the large perf deltas
are mostly the box recovering ‚Äî this round claims **perf-neutral**, not
+21%. e2e green (36/36, was 26), parity `213309Z` = **22 PASS / 2 GAP /
1 N/A** (B2+B3 flipped, zero downgrades). `baseline.json` ‚Üê `213153Z`,
`parity-baseline.json` ‚Üê `213309Z`. Also fixed: `bench/bench.sh`'s
