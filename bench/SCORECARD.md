# icegres Scorecard

Definitive parity + performance record for **icegres** ÔÇö a serve-in-place
Iceberg lakehouse server speaking the Postgres wire protocol ÔÇö measured
against the behavioral bar set by Databricks Lakebase (= Neon architecture)
and Moonlink. Contract: `bench/SPEC.md`. All raw probe/benchmark JSONs live
in `bench/results/`; the parity matrix below is machine-regenerated by
`bench/parity.sh`, benchmark tables are appended by `bench/bench.sh`, and
every improvement landed through the regression gate `bench/gate.sh`
(FAIL if any latency p50 worsens >20%, qps drops >10%, e2e not green, or
any parity verdict downgrades).

## Reproduction (3 commands)

```bash
bash infra/scripts/up.sh    # 1. start/health-check the stack (Postgres :5433, RustFS S3 :9000, Lakekeeper REST :8181)
bash bench/parity.sh        # 2. all 25 parity probes -> bench/results/parity-<ts>.json + regenerates the matrix below
```

appends a few rows (= small Parquet files) to the append-only `demo.trips`;
see **Layout drift** at the bottom before comparing full-scan numbers
across runs.

## GAP roadmap ÔÇö the 6 honest gaps

All six are consequences of two facts: the pgwire front end ships no-op
auth/plaintext transport by default, and the pinned iceberg-rust 0.9.1 /
iceberg-datafusion integration is append-only (fast_append is the only
data-changing transaction action). None can be fixed by icegres-side code
alone without either real security plumbing or an upstream Iceberg
write-path release; each has a concrete roadmap:

| id | gap | why it exists | roadmap |
|----|-----|---------------|---------|
| A6 | no server-side auth | `AuthManager::default()` is a no-op; any user/password connects | icegres already runs its own accept loop (`icegres/src/ops.rs`, built for idle-shutdown); wire pgwire's SCRAM/cleartext `StartupHandler` there against a password file / env secret. Effort: small-medium, no upstream blocker. |
| A7 | no TLS | no rustls acceptor on the listener | short term: terminate TLS in front (stunnel/haproxy/pgbouncer). Native: pgwire supports rustls; plumb cert/key config into the same ops.rs accept loop. No upstream blocker. |
| B2 | UPDATE rejected | iceberg-datafusion 0.9 is append-only; Iceberg mutation needs v2 delete files (MoR) or a rewrite-files action (CoW) ÔÇö neither exists in 0.9.1 | blocked on upstream iceberg-rust; revisit at each release. Do NOT hand-forge REST commits outside the library's conflict detection. |
| B3 | DELETE rejected | same as B2 | same as B2. |
| B4 | BEGIN/COMMIT/ROLLBACK accepted but non-transactional (INSERT inside ROLLBACK persists) | every statement auto-commits an Iceberg snapshot | feasible within 0.9.1 for INSERT-only transactions: buffer appends per session, emit ONE `fast_append` commit at COMMIT, drop buffers at ROLLBACK. Medium effort; honest interim behavior documented by the probe. |
| B5 | no PK/unique constraints (duplicate insert accepted) | Iceberg has no constraint concept | enforcement would need a read-before-write existence check per INSERT ÔÇö expensive and racy across multiple computes (D2). Documented as an architectural anti-goal; dedupe at query time or in future compaction. |

N/A-BY-DESIGN (1): C3 CDC PostgresÔćĺIceberg ÔÇö Moonlink exists to copy
Postgres heap data into Iceberg; icegres data is *born* in Iceberg (each
wire INSERT is an Iceberg commit, proven by B1+C1), so there is no second
copy to synchronize. Parity by construction.

### Layout drift ÔÇö read this before comparing runs

`demo.trips` is append-only and every e2e/parity run adds rows as new small
Parquet files (~1 row/file). `baseline.json` was captured at 284 rows /
5 data files; by this verification pass the table was at 292 rows /
**13 data files** (e2e + parity appends from the fresh pass itself). Each
extra file costs ~0.3ÔÇô1 ms per full scan, which is exactly the observed
aggregate/join delta and the qps drop (mixed-read queries pay it too). The
mechanism was proven during parity-features acceptance: a **control binary
with the change removed showed the same "regression" vs baseline**
(qps Ôłĺ23.5%, aggregate +28.2% at 4 extra files) while candidate-vs-control
at matched layout showed none. Gate rule of thumb: if a gate fails on
qps/aggregate only, benchmark a control build at the same table layout
before blaming the change. Reset path: drop the demo tables via Lakekeeper
REST (`purgeRequested=true`) and re-run `icegres seed` (restores 1 file per
table; documented in `icegres/src/seed.rs`).

## Detailed improvement notes

### 1. metadata-caching (accepted; baseline `163255Z` Ôćĺ `164622Z`)

**Change** (`icegres/src/cache.rs`, wired in `icegres/src/context.rs`):
snapshot-aware table metadata caching. iceberg-datafusion's catalog-backed
provider called `load_table` on every scan and rebuilt a `Table` whose
internal manifest cache starts cold, so every query re-read the manifest
list + all ~231 manifest avro files from RustFS (~1 ms/GET Ôëł the entire
~220 ms point-lookup p50); the Lakekeeper REST call itself is only
~2ÔÇô3 ms. `CachingTableProvider` keeps the last-seen `Table` (warm moka
manifest cache) inside an `IcebergStaticTableProvider` and does one cheap
REST `load_table` per scan purely to compare metadata-location/snapshot-id:
unchanged Ôćĺ reuse warm provider; changed (commit from this server or any
other writer) Ôćĺ rebuild. **Freshness stays exact ÔÇö no TTL, zero staleness
window.** INSERTs delegate to the upstream provider; metadata tables and
DDL fall through.

Gate PASS (e2e green; parity `164753Z` unchanged at 16/8/1):
point_lookup 221.64 Ôćĺ 14.74 (Ôłĺ93.3%), filtered_scan 230.12 Ôćĺ 14.95
(Ôłĺ93.5%), aggregate 253.96 Ôćĺ 121.56 (Ôłĺ52.1%), join 257.80 Ôćĺ 127.39
(Ôłĺ50.6%), qps 7.6 Ôćĺ 28.1 (+269.7%); writes/freshness within gate
(+9.7ÔÇŽ+12.8%, noise).

### 2. datafusion-tuning (accepted; baseline `164622Z` Ôćĺ `171214Z`)

**Measurement first**: interleaved A/B sweeps proved every classic
SessionConfig/RuntimeEnv knob a no-op on this workload; `EXPLAIN ANALYZE`
showed all operator time in ┬Ás with ~all latency in the single-partition
`IcebergTableScan` stream, whose object-store IO is hardwired to
`num_cpus` (= 4) concurrent GETs in iceberg-rust 0.9. The effective
"object-store concurrency" knob lives in the Iceberg scan builder, not
SessionConfig.

**Change**: `icegres/src/scan.rs` ÔÇö `TunedIcebergScan`, a drop-in
`ExecutionPlan` rebuilt from the upstream scan's public getters, running
the identical `TableScan` with `.with_concurrency_limit(N)`;
`N` from `ICEGRES_SCAN_CONCURRENCY` (default 32; `0` = upstream behavior).
Wired in `CachingTableProvider::scan`; inserts/metadata/DDL untouched.
Plus `ICEGRES_DF_OPTS` env passthrough for ops tuning (defaults unchanged
ÔÇö nothing measured an improvement).

Gate PASS (e2e green; parity `171352Z` unchanged at 16/8/1):
point_lookup Ôłĺ20.8%, filtered_scan Ôłĺ18.0%, aggregate Ôłĺ25.8%, join Ôłĺ21.8%,
freshness Ôłĺ14.3%, qps 0% (8 conns already saturate 4 cores), nothing
regressed. Remaining read cost diagnosed as file-count-bound (~236 tiny
Parquet files) Ôćĺ next improvement.

### 3. seed-file-layout (accepted; baseline `171214Z` Ôćĺ `172127Z`)

**Change** (`icegres/src/seed.rs`, `icegres/src/context.rs`): seed writes
each demo table as exactly ONE Parquet file in ONE Iceberg commit ÔÇö
`insert_all` (single 280-row INSERT) instead of 100-row chunks, and the
seed session pins `target_partitions = 1`
(`build_session_context_with(catalog, Some(1))`; iceberg-datafusion writes
one file per non-empty repartition worker, so 1 partition Ôçĺ 1 file
deterministically). serve/sql paths unchanged. Demo tables were dropped
(Lakekeeper REST, `purgeRequested=true`) and re-seeded: `demo.trips` went
from 239 data files to 1. No `icegres compact` ÔÇö see "Rejected along the
way".

Gate PASS (e2e green; parity `172254Z` unchanged at 16/8/1):
aggregate Ôłĺ92.9% (90.18 Ôćĺ 6.39), join Ôłĺ90.5%, filtered_scan Ôłĺ52.0%,
point_lookup Ôłĺ45.5%, qps +1533.8% (28.1 Ôćĺ 459.1), rss Ôłĺ10.8%, writes
Ôłĺ6/Ôłĺ9%. Read metrics are intentionally not layout-comparable across the
re-seed ÔÇö the layout change IS the improvement.

### 4. parity-features (accepted; baseline `172127Z` Ôćĺ `173613Z`, parity baseline Ôćĺ `174423Z`)

**Change** ÔÇö closed the two cheap parity GAPs and hardened E2:

- **D5 scale-to-zero** (`icegres/src/ops.rs`): `icegres serve
  --idle-shutdown-secs N` (env `ICEGRES_IDLE_SHUTDOWN_SECS`) exits cleanly
  (code 0) after N seconds with zero client connections, via an own accept
  loop over `pgwire::tokio::process_socket` (flagless path = stock
  datafusion-postgres loop, byte-for-byte). Probe: served queries, exited
  on its own 5.0 s after the last connection, respawned and answered in
  ~1.3 s Ôćĺ GAP Ôćĺ PASS. Supervisor pattern documented in ops.rs + README.
- **D4 time travel** (`icegres/src/cache.rs`): quoted
  `"table@snapshot_id"` references resolve to a read-only provider pinned
  via `IcebergStaticTableProvider::try_new_from_table_snapshot` (immutable
  Ôćĺ cached forever; unknown ids error; INSERT rejected). Probe: pinned
  oldest snapshot returns 280 rows vs current live count Ôćĺ GAP Ôćĺ PASS.
- **E2 health** (`icegres/src/ops.rs`): `--health-port P` /
  `ICEGRES_HEALTH_PORT` ÔÇö minimal HTTP liveness endpoint (200 "ok");
  health pings deliberately do not reset the idle clock.

Parity 16/8/1 Ôćĺ **18/6/1**, zero downgrades; e2e green. First gate run
FAILed vs the stale baseline ÔÇö proven environmental by control-binary A/B
(see "Layout drift"); candidate vs control at matched layout: aggregate

Two candidate runs agreed within 25% on every metric; `173613Z` became
`baseline.json`.
